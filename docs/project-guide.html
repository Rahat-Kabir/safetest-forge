1. Big picture

What this project does

SafeTest Forge is a TypeScript tool that creates and runs Python pytest tests for an existing local Python repository. The important promise is not just "AI writes tests." The promise is "AI writes tests, the tool runs them, records what happened, blocks unsafe edits, and lets you inspect or rewind the work."

Input

A path to a local Python repo, optional target path, repair limit, timeout, and agent mode.

Work

Analyze Python files, generate tests, run pytest, classify failures, and repair generated tests once or twice.

Output

Generated test files, a structured report, a trace event log, checkpoint metadata, and rewind data.

The project is CLI-first. Every important feature must work from the CLI before the UI wraps it. The React UI is a dashboard over the same run engine, not a separate implementation.
2. Architecture

One engine, two front doors

The CLI and UI both call the same RunService. That service is the center of the application: it validates the repo, asks an agent to generate tests, runs pytest, records events, and writes the final report.

CLI: command entry point Server: local API RunService: orchestration Agent: fake or Claude Runtime: Python and pytest Store: JSON and NDJSON files

4. CLI, UI, fake, Claude

Four names you will see constantly

There are two interfaces and two agent modes. The interfaces choose how you operate the tool. The agent mode chooses who writes the generated tests.

CLI

Best for learning, automation, and seeing raw output. It calls RunService directly.

npm run cli -- run --repo tests/fixtures/simple-package --agent-mode fake
npm run cli -- report --run <run-id>
npm run cli -- trace --run <run-id>
npm run cli -- rewind --run <run-id>

UI

Best for visually watching a run. It talks to the Express server through REST and listens to live trace events through SSE.

npm run build:ui
npm run server

open http://127.0.0.1:4317

fake mode: safe learning path

FakeAgentClient writes predictable tests based on fixture source code. It costs nothing, needs no API key, and makes the integration tests deterministic.

Use this first whenever you want to understand the lifecycle.

claude mode: real agent path

ClaudeAgentClient uses @anthropic-ai/claude-agent-sdk. It builds prompts, defines subagents, enables file checkpointing, normalizes SDK messages, and applies policy hooks before file writes or shell commands.

This requires ANTHROPIC_API_KEY and can spend API credits.

server mode: the UI backend

npm run server starts Express on 127.0.0.1:4317. It writes a local token to .safetest-forge/server.json and requires that token on mutating endpoints.

The UI page receives the token when the server renders index.html.

5. Storage

Every run leaves a local paper trail

The project uses flat files under .safetest-forge/. That makes it easy to inspect what happened without a database.

run.json Run config and status: repo path, target path, agent mode, timeout, generated files.
events.ndjson One normalized trace event per line. The UI streams and renders these events.
report.json Final structured result: modules, tests, pytest summary, repair, cost, checkpoints.
control.json Cancellation state for a run. The CLI and API can request cancellation.
rewind-snapshot.json Snapshot data used by fake mode and fallback rewind behavior.
A trace is not the same as a report. The trace is the timeline of what happened. The report is the final answer after the run settles.
6. Safety model

The agent is boxed into test work

V1 is designed to generate tests, not edit production code. The policy layer checks writes and shell commands. In Claude mode, these checks are enforced through SDK PreToolUse hooks.

Allowed write paths

  • Files inside tests/
  • Python files named like test_*.py
  • Only inside the selected repository

Blocked behavior

  • Writing production files such as src/foo.py
  • Dangerous commands like rm -rf
  • Broad shell commands outside the V1 allowlist
7. Try it for real

A beginner-friendly lab

Start in fake mode. Then inspect the generated file, report, and trace. Only after that should you try Claude mode.

Fake run: no API cost

npm install
npm run build
npm run cli -- run --repo tests/fixtures/simple-package --agent-mode fake
  • 1Copy the printed run_id.
  • 2Open tests/fixtures/simple-package/tests/test_calculator.py.
  • 3Inspect the final report with npm run cli -- report --run <run-id>.
  • 4Inspect the event stream with npm run cli -- trace --run <run-id>.

Claude run: real model path

# .env must include:
ANTHROPIC_API_KEY=your_key_here

npm run cli -- run --repo tests/fixtures/simple-package --agent-mode claude

Claude mode uses the real Agent SDK, subagents, checkpointing, hooks, structured output, and API cost tracking. Use it when you want to test actual model behavior.

Tip: add --budget-limit-usd 1 for a cautious live test.

UI test

npm run build:ui
npm run server

# Browser:
http://127.0.0.1:4317

In the UI, enter a repo path such as D:\PROJECT\safetest-forge\tests\fixtures\simple-package. The server reads the default agent mode from .env, so be aware of whether your environment is set to fake or claude.

8. Code map

Where to look when you are learning

Read the project in this order. It follows the run path from the outside toward the core.

Start here

  • README.md: commands and capabilities
  • package.json: scripts and dependencies
  • src/cli.ts: CLI command definitions
  • src/run-service.ts: the main orchestration brain

Then follow the helpers

  • src/runtime/python.ts: repo scan, pytest, coverage, failure classification
  • src/agent/fake-agent-client.ts: deterministic local agent
  • src/agent/claude-agent-client.ts: real SDK integration
  • src/policy/policy.ts: write and shell safety

UI and API

  • src/server/app.ts: REST endpoints and SSE stream
  • src/ui/App.tsx: run panel, files panel, report panel, trace panel
  • scripts/build-ui.mjs: bundles React UI into dist/ui

Contracts and persistence

  • src/types.ts: shared TypeScript shapes
  • src/trace/normalize.ts: SDK events -> app events
  • src/storage/run-store.ts: flat-file persistence
  • prompts/*.md: agent instructions
9. Glossary

Terms without the fog

Search the terms you see in reports, traces, and code.

RunOne complete attempt to analyze a repo, generate tests, run pytest, and save a report.
Run IDA UUID used to find one run's report, trace, and metadata.
Trace eventA normalized event such as tool_use, file_changed, or run_finished.
ReportThe final JSON summary of status, generated tests, pytest results, repair, checkpoints, and cost.
Fake agentA deterministic local test writer used for development and test coverage without API calls.
Claude agentThe real Agent SDK client that can inspect files, write tests, and emit rich streaming events.
Policy hookA pre-tool-use guard that allows or denies file writes and shell commands.
CheckpointA restore point captured during generation so file changes can be rewound later.
Repair loopA small retry step that edits generated tests only when failure looks repairable.
SSEServer-sent events. The UI uses it to receive live trace events from the backend.
pytest-json-reportOptional Python plugin that lets the runtime capture per-test case results.
pytest-covOptional Python plugin that lets the runtime capture line coverage summaries.
